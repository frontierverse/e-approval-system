# 결재온 UI/UX 개발 가이드라인

## 1. 목적

결재온은 사용자가 매일 반복적으로 사용하는 사내 업무 시스템이다. 화면의 목적은
멋있어 보이는 여백을 만드는 것이 아니라, 사용자가 사이트를 연 직후 현재 업무량과
우선순위, 다음 행동을 빠르게 파악하도록 돕는 것이다.

핵심 원칙은 다음 한 문장으로 정리한다.

> 실제 정보량 때문에 생기는 스크롤은 허용하지만, 장식·중복 설명·과도한 여백 때문에
> 생기는 스크롤은 UX 결함으로 본다.

이 문서는 신규 화면, 기존 화면 개선, 로딩·빈 상태, 반응형 화면을 포함한 모든 UI 작업의
기본 승인 기준이다.

UI 검토나 개선을 요청받으면 이 기준에 따라 문제를 직접 찾고, 안전한 범위의 개선을 구현하고,
검증까지 진행한다. 일반적인 간격·배치·컴포넌트 선택을 다시 사용자에게 묻지 않는다. 결과를
크게 바꾸는 업무 우선순위나 정책 결정이 비어 있을 때만 확인한다.

## 2. 의사결정 우선순위

UI 선택이 충돌할 때는 다음 순서로 판단한다.

1. 업무를 정확하게 처리할 수 있는가
2. 긴급도와 다음 행동을 빠르게 파악할 수 있는가
3. 한 화면에서 충분한 정보를 비교할 수 있는가
4. 실수를 예방하고 상태를 명확히 전달하는가
5. 접근성과 반응형 사용성이 유지되는가
6. 시각적으로 일관되고 전문적인가
7. 장식적으로 여유롭거나 인상적인가

정보 밀도는 무조건 작게 만드는 것을 뜻하지 않는다. 읽기 어렵게 압축하거나 행동을
숨기는 것도 실패다. 목표는 `최소 공간에 최대 요소`가 아니라 `최소 탐색으로 최대 이해`다.

## 3. 첫 화면 정보 원칙

### 대시보드와 목록 화면

- 사용자가 5초 안에 전체 건수, 긴급 항목, 현재 상태, 주 행동을 찾을 수 있어야 한다.
- 제목·날짜·주 행동은 가능한 한 하나의 짧은 헤더 영역에 배치한다.
- 핵심 요약 지표 다음에 주 업무 목록이 바로 시작되어야 한다.
- 주 업무 목록의 패널 헤더는 가능한 경우 전역 고정·sticky 셸을 제외한 스크롤 가능 콘텐츠
  뷰포트 상단 40% 이내에서 시작하고, 첫 데이터 행이 바로 이어져야 한다.
- 데스크톱에서는 요약 지표, 가장 중요한 업무 목록, 보조 진행 정보가 첫 화면에서 함께
  보이도록 구성한다.
- 모바일에서는 요약 지표와 가장 중요한 업무 목록을 먼저 보여주고, 진행 정보와 이력은
  그 뒤에 배치한다.
- 공지, 변경 기록, 참고 설명처럼 즉시 행동과 관계없는 콘텐츠는 핵심 업무 이후에 둔다.

40% 기준은 대시보드와 목록 화면의 목표다. 폼, 문서 상세, 달력, 시간표, 인쇄 화면처럼
정확한 입력·문맥·공간 관계가 중요한 화면에는 기계적으로 적용하지 않는다. 이런 화면은
오류 예방과 가독성을 우선하되, 중복 설명과 근거 없는 여백은 동일하게 제거한다.

### 피해야 할 패턴

- 큰 환영 문구, 히어로 영역, 의미 없는 일러스트
- `eyebrow + 제목 + 긴 설명 + 건수`를 모든 패널에서 반복하는 구조
- 숫자 하나를 보여주기 위해 큰 세로 카드 공간을 사용하는 구조
- 짧은 빈 상태를 `min-height: 256px` 이상으로 늘리는 구조
- 동일한 정보를 상단 요약과 본문에서 같은 문장으로 반복하는 구조
- 첫 실제 업무 항목을 보기 위해 소개 영역부터 스크롤해야 하는 구조

## 4. 기본 밀도 기준

다음 값은 시작점이며, 콘텐츠 특성상 더 많은 공간이 필요하면 명확한 이유가 있어야 한다.

| 요소 | 권장 기준 |
| --- | --- |
| 기본 간격 | 4, 8, 12, 16, 24px 체계 사용 |
| 패널 사이 간격 | 기본 12~16px, 큰 의미 전환만 24px |
| KPI/요약 카드 | 약 64~80px 높이 |
| 패널 헤더 | 약 48~56px 높이 |
| 반복 목록 행 | 데스크톱 약 56~72px, 모바일 약 64~88px |
| 빈 상태 | 보통 120~160px 이내 |
| 클릭·터치 영역 | 최소 44×44px |

- 터치 영역 44px는 버튼 주변의 거대한 카드 여백을 의미하지 않는다. 시각적 높이와 실제
  상호작용 영역을 구분해 설계한다.
- `mt-6`, `py-8`, 큰 `min-height`를 관성적으로 사용하지 않는다. 각 간격이 정보 그룹을
  구분하는 데 실제로 필요한지 확인한다.
- 반복 목록에서 아바타, 설명, 배지, 구분선을 모두 사용하지 않는다. 업무 판단에 필요한
  필드만 남긴다.
- 숫자, 건수, 날짜, 단계는 `tabular-nums`를 사용해 빠르게 비교할 수 있게 한다.

## 5. 컴포넌트 선택

### 카드가 적합한 경우

- 서로 다른 종류의 지표를 비교할 때
- 독립된 업무 그룹을 구분할 때
- 상태나 행동이 항목마다 크게 다를 때

### 행·목록·표가 적합한 경우

- 같은 형태의 문서나 기록이 반복될 때
- 제목, 담당자, 대기 시간, 상태, 단계처럼 동일 필드를 비교할 때
- 한 화면에서 여러 항목을 훑어봐야 할 때

반복 데이터를 큰 카드 묶음으로 만들지 않는다. 데스크톱에서는 열 정렬을 활용하고,
모바일에서는 같은 정보를 중요도 순으로 2~3줄에 재배치한다.

## 6. 정보 계층과 행동

- 한 화면의 주 행동은 하나를 가장 명확하게 강조한다.
- 위험·지연·반려 등 즉시 확인이 필요한 상태만 강한 의미 색상을 사용한다.
- 제목, 문서번호, 담당자, 대기 시간, 결재 단계가 충돌하면 제목과 긴급도를 먼저 보존한다.
- 공간이 부족하면 낮은 우선순위 메타데이터를 숨기거나 다음 줄로 보내되, 핵심 상태와
  접근 가능한 이름은 유지한다.
- `전체 보기`처럼 문맥에 의존하는 링크는 `aria-label`로 대상을 명확히 한다.
- 행 전체가 링크라면 별도의 큰 `보기` 버튼을 반복하지 않는다. 대신 hover, focus, 방향
  표시로 클릭 가능성을 전달한다.
- 진행률은 단계 숫자와 함께 제공한다. 색상이나 막대만으로 상태를 전달하지 않는다.

## 7. 반응형 기준

- 데스크톱은 넓은 화면을 단순히 여백으로 남기지 말고, 주 목록과 보조 상태 영역을 동시에
  비교하는 데 사용한다.
- 모바일은 데스크톱을 그대로 세로로 쌓는 것이 아니라 업무 우선순위에 따라 재정렬한다.
- 모바일 기본 순서는 `제목/주 행동 -> 핵심 지표 -> 처리할 업무 -> 내 진행 -> 최근 이력`이다.
- 핵심 지표는 모바일에서 보통 2열, 데스크톱에서 4열을 우선 검토한다.
- 모바일에서 부서·직급 같은 보조 정보는 필요하면 축약하지만, 제목·긴급도·대기 시간·단계는
  우선 보존한다.
- 가로 스크롤은 의도된 표·타임라인이 아닌 이상 허용하지 않는다. 허용하는 경우 첫 열이나
  핵심 식별 정보를 보존하고, 스크롤 가능성을 시각적으로 알리며, 키보드로도 모든 내용을
  탐색할 수 있어야 한다.
- 작은 화면에서 버튼이 전체 너비를 차지해 헤더 높이를 키우지 않도록, 짧은 제목과 행동은
  가능한 경우 한 행에 배치한다.

## 8. 사내 시스템의 시각 언어

- 차분하고 절제된 표면, 얇은 테두리, 일관된 모서리와 간격을 사용한다.
- 브랜드 색상은 선택, 주 행동, 진행 상태를 강조하는 데 제한적으로 사용한다.
- 위험 색상은 실제 위험·지연·실패 의미에만 사용한다.
- 불필요한 그라데이션, 과도한 그림자, 떠오르는 애니메이션, 장식 아이콘을 추가하지 않는다.
- hover 효과는 위치가 크게 움직이지 않도록 하고, 색상·테두리·약한 그림자 중심으로 표현한다.
- 라이트·다크 모드 모두 기존 CSS 변수와 디자인 토큰을 사용한다. 새 고정 색상을 추가할 때는
  두 테마의 대비를 함께 검증한다.

## 9. 상태 화면

### 로딩

- 스켈레톤은 실제 최종 레이아웃과 같은 열, 행 수, 높이를 사용해 레이아웃 이동을 줄인다.
- 스켈레톤 자체가 실제 화면보다 크거나 느슨하면 안 된다.
- `prefers-reduced-motion`에서는 반복 애니메이션을 줄인다.

### 빈 상태

- 빈 상태는 짧은 제목, 한 줄 설명, 필요한 경우 하나의 다음 행동만 제공한다.
- 빈 공간을 채우기 위해 큰 아이콘이나 높은 최소 높이를 사용하지 않는다.
- 정상적인 완료 상태와 오류 상태를 시각적으로 구분한다.

### 오류와 완료 피드백

- 무엇이 실패했는지와 사용자가 다음에 할 수 있는 행동을 함께 알려준다.
- 성공 피드백 때문에 사용 흐름이 중단되거나 정보가 가려지지 않게 한다.

## 10. 업무 흐름과 안전성

- 저장·제출·승인·반려처럼 서버 상태를 바꾸는 행동은 처리 중 다시 실행할 수 없게 해 중복
  요청을 방지한다.
- 검증 실패나 서버 오류가 발생해도 사용자가 입력한 내용을 가능한 한 보존한다.
- 오류 메시지는 원인과 복구 행동을 알려주고, 첫 오류 또는 오류 요약으로 포커스를 이동한다.
- 삭제, 회수, 반려, 권한 변경처럼 영향이 큰 작업은 대상과 결과를 명확히 적은 확인 단계를
  제공한다. 가능하면 복구 가능 여부도 알린다.
- 권한이 없는 행동은 서버에서 반드시 차단하며, UI에서도 숨기거나 비활성 이유를 명확히
  전달한다.
- 민감정보와 권한 전용 데이터는 CSS로만 숨기지 않고, 권한이 없는 사용자에게 렌더링하거나
  전송하지 않는다.
- 비동기 작업의 처리 중·성공·실패 상태를 명확히 보여주되, 피드백이 다음 업무를 가리지
  않게 한다.

## 11. 접근성

- 페이지당 명확한 `h1` 하나와 논리적인 제목 계층을 사용한다.
- 목록, 표, 탐색, 보조 영역은 의미에 맞는 HTML 요소와 접근 가능한 이름을 사용한다.
- 모든 상호작용 요소에 키보드 포커스가 보여야 한다.
- 색상만으로 상태를 전달하지 않는다.
- 일반 텍스트는 최소 4.5:1, 큰 텍스트와 UI 경계·상태 표시는 최소 3:1 대비를 목표로 하며,
  라이트·다크 모드에서 모두 확인한다.
- 클릭·터치 대상은 최소 44×44px를 확보한다.
- 숨긴 보조 텍스트가 접근 이름을 불필요하게 장황하게 만들지 않는지 확인한다.
- 애니메이션은 `prefers-reduced-motion`을 존중한다.
- 문서 순서와 키보드 탭 순서가 시각적 업무 순서와 일치해야 한다.
- 모달은 열릴 때 적절한 위치로 포커스를 이동하고, 내부에서 포커스를 관리하며, 닫힌 뒤
  실행한 요소로 포커스를 돌려보낸다. `Escape`로 닫을 수 있어야 한다.
- 브라우저 200% 확대에서도 핵심 정보와 행동이 사라지거나 겹치지 않아야 한다.

## 12. 검수 기준

레이아웃·반응형·내비게이션·상태·색상에 실질적인 영향을 주는 UI 작업은 실제 브라우저에서
영향받는 화면과 상태를 확인한다. 단순 문구나 레이아웃에 영향이 없는 작은 변경은 영향 범위에
비례한 정적·자동 검수로 대체할 수 있다.

기본 화면 크기:

- 데스크톱 기준: 1366×768
- 넓은 데스크톱 참고: 1440×900
- 모바일 기준: 390×844
- 작은 모바일 영향이 큰 작업: 360×800 또는 320px 너비 추가 확인

검수 항목:

1. 첫 화면에서 건수, 긴급 항목, 주 행동이 보이는가
2. 소개나 공백 때문에 핵심 목록이 접히지 않는가
3. 동일 정보가 여러 번 반복되지 않는가
4. 긴 제목과 큰 숫자에서 레이아웃이 깨지지 않는가
5. 가로 넘침이나 의도하지 않은 내부 스크롤이 없는가
6. 모바일에서 정보 우선순위가 유지되는가
7. 키보드 포커스와 접근 가능한 이름이 명확한가
8. 로딩·빈 상태가 최종 화면보다 과도하게 크지 않은가
9. 색상 변경이 있으면 라이트·다크 모드가 모두 읽히는가
10. 200% 확대와 키보드 탭 순서에서 정보와 행동이 유지되는가
11. 모달을 변경했다면 포커스 진입·내부 이동·닫기·복귀가 정상인가
12. 상태 변경 행동을 수정했다면 중복 실행 방지와 오류 후 입력 보존이 되는가

대표 데이터는 영향 범위에 따라 `0건`, `1건`, 다수·대량, 긴 한글 제목, 큰 숫자, 여러 상태
배지, 일반 사용자와 관리자 권한을 확인한다. 모든 조합을 매번 확인하는 것이 아니라 변경으로
깨질 가능성이 있는 경계값을 선택한다.

실화면 검수가 필요한 작업의 완료 보고에는 `검수 경로`, `뷰포트`, `첫 화면에서 확인된 정보`,
`가로 넘침 여부`를 구체적으로 적는다. 실화면을 확인하지 않고 `한 화면에 보인다`고 추정하지
않는다.

## 13. 완료 조건

UI 변경은 다음 조건을 충족해야 완료로 본다.

- 변경 영향에 비례해 관련 브라우저 화면, 상태, 뷰포트를 확인했다.
- 핵심 업무가 장식적 요소보다 먼저 보인다.
- 불필요한 설명, 여백, 중복 버튼을 제거했다.
- 접근성 의미와 최소 상호작용 영역을 유지했다.
- 변경이 로딩·빈 상태에 영향을 준다면 해당 상태도 함께 점검했다.
- 관련 테스트와 ESLint를 통과했다.
- 배포에 영향을 주는 변경은 프로덕션 빌드까지 통과했다.

예외가 필요하면 가독성, 오류 예방, 법적·운영 요구처럼 공간을 더 사용해야 하는 이유를 코드
또는 변경 설명에 남긴다.
